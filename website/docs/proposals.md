# Feature Proposals

Feature proposals for improving sites generated by eigen.

## Performance & Loading

### 1. Critical CSS Inlining

Extract above-the-fold CSS and inline it in `<head>`, deferring the rest. Reduces render-blocking resources and improves Largest Contentful Paint (LCP).

**Approach**: Parse rendered HTML to identify used selectors in the viewport, inline those styles, and load the full stylesheet asynchronously via `<link rel="preload" as="style">`.

### 2. Preload / Prefetch Hints

Auto-generate `<link rel="preload">` for hero images and `<link rel="prefetch">` for likely next navigations (based on the sitemap or a config hint). Makes HTMX partial loads near-instant for predicted navigations.

**Approach**: Add frontmatter fields (`preload: ["/images/hero.jpg"]`) and a config-level option to prefetch all internal links or a curated list.

### 3. Content Hashing for Static Assets

Fingerprint CSS/JS/images with a content hash in the filename (`style.a1b2c3.css`) and rewrite all references. Enables aggressive `Cache-Control: immutable` headers.

**Approach**: After copying `static/` to `dist/`, hash each file, rename it, and rewrite references in rendered HTML and CSS. Expose the manifest to templates via an `asset_hash(path)` function.

### 4. Lazy Loading for Below-Fold Images

When rewriting `<img>` to `<picture>`, add `loading="lazy"` and `decoding="async"` for images not marked as hero/above-fold. Natural extension of the existing responsive image pipeline.

**Approach**: Default all images to lazy loading. Allow opt-out via a class or attribute (e.g., `data-eager`). First image on a page could auto-detect as eager.

### 5. CSS/JS Bundling and Tree-Shaking

Combine multiple `<link>` / `<script>` references from `static/` into single files. Dead-code-eliminate unused CSS selectors against rendered HTML.

**Approach**: Use `lightningcss` for CSS processing and bundling. Collect all referenced stylesheets per page (or globally), merge them, and remove selectors that don't match any element in the rendered output.

## SEO & Social

### 6. Open Graph / Twitter Card Meta Tags

Allow frontmatter fields like `title`, `description`, `image`, `type` and auto-inject `<meta property="og:...">` and `<meta name="twitter:...">` tags into `<head>`.

**Approach**: Define an `[seo]` section in frontmatter. During rendering, inject the meta tags before `</head>`. Provide sensible defaults from `site.toml` (`site.name` as fallback title, etc.).

### 7. Structured Data (JSON-LD)

Generate `<script type="application/ld+json">` blocks from frontmatter. Support `Article`, `BreadcrumbList`, `WebSite` schemas out of the box.

**Approach**: Add a `schema` frontmatter field that maps to schema.org types. Eigen auto-populates fields from page context (URL, title, dates, images) and injects the JSON-LD block.

### 8. Canonical URL Tags

Auto-insert `<link rel="canonical">` using `base_url` + page path. Prevents duplicate content issues with search engines.

**Approach**: Inject automatically for every page. Allow frontmatter override (`canonical: https://other.example.com/original`).

### 9. RSS/Atom Feed Generation

For dynamic collections (blog posts, etc.), auto-generate `feed.xml`. Configurable per-collection in frontmatter or `site.toml`.

**Approach**: Add `[feed]` config in `site.toml` specifying which collections get feeds. Generate valid Atom XML with title, link, updated, and per-entry content (truncated or full). Write to `dist/feed.xml` (or per-collection paths).

### 10. robots.txt Generation

Simple config-driven `robots.txt` with sitemap reference.

**Approach**: Add `[robots]` section in `site.toml` with allow/disallow rules. Auto-reference the generated sitemap URL.

## Developer & Author Experience

### 11. Incremental Builds

Track file hashes and only re-render pages whose templates, data, or dependencies changed. The smart rebuild logic in the dev server already does scope detection -- extend it to production builds.

**Approach**: Maintain a build manifest (`.eigen_cache/manifest.json`) mapping each output file to the hashes of its inputs (template, layout chain, data files, referenced assets). Skip pages with unchanged inputs.

### 12. Template Dependency Graph

Track which layouts/partials each page uses. When a partial changes, only rebuild affected pages instead of all pages.

**Approach**: Instrument Minijinja's template loading to record which templates are loaded during each page render. Store the dependency graph in the build manifest.

### 13. Build-Time Include with Glob

Support `{% include "components/*.html" %}` to include all matching partials. Useful for icon sprite sheets, component libraries.

**Approach**: Register a custom Minijinja function or extend the include mechanism to resolve glob patterns against the template directory.

### 14. Draft / Scheduled Pages

Frontmatter `draft: true` or `publish_date: 2026-04-01` to exclude pages from production builds but show them in dev.

**Approach**: During discovery, check frontmatter for draft/publish fields. In production builds, skip pages that are drafts or have a future publish date. In dev mode, render them with a visual indicator.

### 15. Pagination

For collections, generate `/posts/page/1.html`, `/posts/page/2.html` etc. with `page.next_url`, `page.prev_url`, `page.total_pages` in context.

**Approach**: Add `paginate: 10` to collection frontmatter. During dynamic page rendering, chunk the collection and render a page per chunk. Provide pagination metadata in the template context.

## Content & Templating

### 16. Markdown Content Files

Support `content/posts/hello.md` alongside templates. The template defines layout, the `.md` file provides body + frontmatter.

**Approach**: Add a `content/` directory convention. During discovery, pair `.md` files with their layout template (specified in frontmatter). Render markdown to HTML, then inject into the template as `{{ content }}`.

### 17. Table of Contents Generation

A `toc` filter or function that extracts headings from markdown content and returns a structured list.

**Approach**: After markdown rendering, parse the HTML for `<h1>`-`<h6>` tags, build a nested structure with `id`, `text`, and `level`, and expose it as `{{ content | toc }}` or `{{ toc(content) }}`.

### 18. Shortcodes / Inline Components

Custom tags inside markdown like `{{< youtube id="abc123" >}}` that expand to HTML snippets.

**Approach**: Pre-process markdown content before rendering, replacing shortcode patterns with rendered partials from a `_shortcodes/` directory. Pass shortcode attributes as template context.

### 19. i18n / Multilingual Support

Config-driven locale routing (`/en/about.html`, `/fr/about.html`) with a `t("key")` template function that reads from locale YAML files. Generates `hreflang` tags automatically.

**Approach**: Add `[i18n]` config with `locales = ["en", "fr"]` and `default = "en"`. Load `_locales/en.yaml` etc. Render each page once per locale, prefixing output paths. Inject `hreflang` link tags.

### 20. Related Content Function

A `related_to(item, collection, fields, limit)` template function that scores similarity across specified fields.

**Approach**: Implement a simple TF-IDF or tag-overlap scorer. During rendering, compute similarity between the current item and all collection items across the specified fields. Return the top N matches.

## HTMX & Interactivity

### 21. View Transitions API Integration

Add `<meta name="view-transition" content="same-origin">` and CSS `view-transition-name` attributes to fragment containers. Gives smooth animated transitions between partial loads with zero JS.

**Approach**: Add a `view_transitions: true` config option. When enabled, inject the meta tag and add `view-transition-name` to fragment target elements. Provide a default CSS transition and allow customization.

### 22. Prefetch on Hover / Viewport

Add `hx-trigger="mouseenter"` or `hx-trigger="intersect"` preloading for links. HTMX supports this natively.

**Approach**: Extend `link_to` with a `prefetch` parameter. When set, emit `hx-trigger="mouseenter once"` with a preload extension, or use HTMX's built-in `preload` extension.

### 23. Fragment-Level Cache Headers

Generate a manifest or `_headers` file with per-fragment cache rules. Fragments for mostly-static content (footer, nav) can be cached aggressively.

**Approach**: Add `cache` config per OOB block. Generate platform-specific header files (`_headers` for Netlify/Cloudflare Pages) mapping fragment paths to cache durations.

### 24. Scroll Restoration Hints

Inject `hx-swap="innerHTML show:window:top"` or similar scroll position management into `link_to` output.

**Approach**: Add a `scroll` parameter to `link_to` (default: `top`). Map to appropriate `hx-swap` modifiers. Allow global default in `site.toml`.

## Security & Headers

### 25. Security Headers File Generation

Generate `_headers` (Netlify/Cloudflare) or `.htaccess` with CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy. Configurable in `site.toml`.

**Approach**: Add `[headers]` config section. Generate the appropriate file format based on a `platform` setting or generate a generic format that can be adapted.

### 26. Subresource Integrity (SRI)

For CDN-loaded scripts/styles that aren't localized (in the skip list), compute and add `integrity="sha384-..."` attributes.

**Approach**: During HTML rewriting, fetch CDN resources, compute SHA-384 hash, and add `integrity` and `crossorigin="anonymous"` attributes to the tags.

---

## Accessibility

### 27. Alt Text Validation

Warn at build time if any `<img>` lacks an `alt` attribute. Optionally fail the build in strict mode.

**Approach**: After rendering, scan HTML for `<img>` tags without `alt`. Log warnings. Add `[accessibility] strict = true` config to make it a build error.

### 28. Skip Navigation Link Injection

Auto-add a "skip to content" link before the first `<nav>` if one doesn't exist.

**Approach**: During HTML post-processing, check for existing skip links. If absent, inject `<a href="#content" class="skip-link">Skip to content</a>` and ensure the content block has `id="content"`.

---

## Deployment

### 29. Redirect Rules from Config

Define redirects in `site.toml` and generate platform-specific files (`_redirects` for Netlify, `staticwebapp.config.json` for Azure, etc.).

**Approach**: Add `[[redirects]]` array in `site.toml` with `from`, `to`, `status` fields. Generate the appropriate file for the configured deployment platform.

### 30. 404 Page Support

Recognize `404.html` template specially and ensure it's generated at the root. Many CDNs/hosts look for this by convention.

**Approach**: During discovery, treat `404.html` as a special static page. Ensure it's output at `dist/404.html` regardless of directory structure. Optionally generate platform-specific error page config.
