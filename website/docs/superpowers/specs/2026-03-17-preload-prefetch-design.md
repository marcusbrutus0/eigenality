# Preload / Prefetch Hints -- Design Spec

Date: 2026-03-17

## Motivation and Goals

Two performance bottlenecks remain after critical CSS inlining:

1. **Hero images block LCP.** The browser discovers the hero image only
   after parsing the HTML and building the DOM. A `<link rel="preload">`
   hint in `<head>` tells the browser to start fetching the image
   immediately, before the HTML parser reaches the `<img>` tag. This
   directly reduces Largest Contentful Paint (LCP) -- the single most
   impactful Core Web Vital for perceived performance.

2. **HTMX navigations are not instant.** When a user clicks a link built
   with `link_to()`, HTMX fetches the fragment file from the server.
   Even with a fast server, there is a visible delay while the browser
   opens the connection and downloads the fragment. A
   `<link rel="prefetch">` hint tells the browser to fetch the fragment
   file at idle time, so it is already in the browser cache when the
   user clicks. This makes HTMX partial-page transitions feel instant.

**Goals:**

- Auto-generate `<link rel="preload">` for hero images designated in
  frontmatter, with correct `as`, `type`, and `imagesrcset`/`imagesizes`
  attributes when responsive images are in use.
- Auto-generate `<link rel="prefetch">` hints for internal navigation
  links, targeting fragment files (not full pages) to match HTMX's
  fetch behavior.
- Provide site-level configuration to enable/disable/tune prefetching.
- Provide per-page frontmatter fields for hero image designation.
- Integrate cleanly into the existing build pipeline with zero impact
  when disabled.

**Non-goals (see "Out of Scope"):**

- Intersection Observer-based lazy prefetching (requires client-side JS).
- Preloading fonts (separate concern; typically done manually in templates).
- Predictive prefetching based on analytics or navigation patterns.
- Preloading external/CDN resources.

## Architecture

### Two Distinct Concerns, One Module

Preload (hero images) and prefetch (navigation) are conceptually related
-- both inject `<link>` hints into `<head>` -- but solve different
problems:

- **Preload** makes the *current page* load faster by fetching a known
  critical resource earlier. The resource URL comes from frontmatter
  (`hero_image`) or from scanning the rendered HTML for the first
  significant `<img>`.
- **Prefetch** makes *future navigations* faster by speculatively
  fetching resources at idle time. The resource URLs come from scanning
  the rendered HTML for `hx-get` attributes (which point to fragment
  files).

Despite their differences, both produce `<link>` tags that are injected
into `<head>` via a single `lol_html` rewriting pass. They live in one
module: `build::hints`.

**Note on lol_html passes:** The module uses up to two `lol_html`
passes: one to scan the HTML for auto-detection data (hero image,
`hx-get` URLs, existing preload hints), and one to inject the generated
`<link>` tags into `<head>`. This is consistent with how the codebase
uses `lol_html` elsewhere (e.g., `assets::html_rewrite` uses separate
collection and rewriting passes). Alternatively, if the frontmatter
`hero_image` is set and we can avoid auto-detection scanning, the
collection pass can be simplified or skipped for the hero portion.

**Why not the plugin system?** Same rationale as critical CSS: resource
hints need access to the rendered HTML structure, the dist_dir
filesystem (to check which images exist and whether they have responsive
variants), and the build config. They are a core performance
optimization that every site benefits from, not an extension point.

### Hero Image Preload Strategy

There are two ways to identify the hero image:

1. **Explicit frontmatter field** (`hero_image: /assets/hero.jpg`).
   The user declares which image is the hero. This is the most reliable
   approach because:
   - The user knows which image is above the fold.
   - The image path is known before rendering, so the preload hint can
     be generated with certainty.
   - It works even when the image is set via CSS `background-image`.

2. **Auto-detection from rendered HTML** (first `<img>` in the content
   area). This is a fallback when no frontmatter field is set:
   - Scan the rendered HTML for the first `<img>` tag that is not
     `loading="lazy"` and not excluded by pattern.
   - This heuristic works well for typical blog/content layouts where
     the first image is the hero.

**Decision:** Support both. Frontmatter `hero_image` takes priority.
When absent and auto-detection is enabled in config, fall back to
scanning the HTML for the first qualifying `<img>`. Auto-detection
defaults to enabled.

When image optimization is active and the hero image has responsive
variants (generated by `assets::images`), the preload hint uses
`imagesrcset` and `imagesizes` attributes so the browser preloads the
correct variant for the viewport width. This is critical because
preloading the wrong size wastes bandwidth.

### Responsive Variant Discovery

Image optimization generates variant files with the naming pattern
`{stem}-{width}w-{hash}.{format}` (e.g., `hero-480w-a1b2c3d4.avif`).
However, by the time hints run, the `<img>` tags have already been
rewritten to `<picture>` elements with `<source srcset="...">` tags
that contain the exact variant URLs.

**Decision:** When auto-detecting from rendered HTML, extract the
`srcset` and `type` directly from the `<picture>` element's first
`<source>` child. This is more reliable than re-scanning the
filesystem because:

1. It uses the exact URLs that the browser will use (including content
   hashes).
2. It avoids duplicating the variant naming logic from
   `assets::images`.
3. The `<source>` elements are already ordered best-format-first
   (AVIF, then WebP, then original) by `determine_format_order` in
   `assets::html_rewrite`.

When using a frontmatter `hero_image` path, we look for a `<picture>`
element whose `<img>` child has a matching `src` attribute. If found,
we extract `srcset` from the first `<source>`. If no matching
`<picture>` is found (e.g., the image was not optimized, or is a CSS
background image), we fall back to scanning dist_dir for variant files
matching `{stem}-*w-*.{format}` in the same directory.

### Navigation Prefetch Strategy

The key insight is that HTMX navigations fetch *fragment files*, not
full pages. The `link_to()` function generates `hx-get` attributes
pointing to fragment paths like `/_fragments/about.html`. Prefetch
hints should target these same fragment URLs.

**Approach:** Scan the rendered HTML for all `hx-get` attributes. Each
unique `hx-get` URL that points to a local path (starts with `/`) gets
a `<link rel="prefetch">` hint in `<head>`.

**Why `hx-get` and not `<a href>`?** An `<a>` tag's `href` points to
the full page, but HTMX fetches the fragment via `hx-get`. Prefetching
the full page would waste bandwidth since HTMX will not use it.
Prefetching the fragment is both smaller and matches exactly what the
browser will fetch on click.

**Limiting prefetch count:** A page with many navigation links (e.g.,
a blog index with 50 posts) should not prefetch all 50 fragments. The
config provides a `max_prefetch` limit (default: 5) that caps the
number of `<link rel="prefetch">` hints per page. Links are included
in document order (first N `hx-get` URLs found), on the assumption
that earlier links are more likely to be clicked (main navigation,
featured content).

**Self-referencing links:** If a page links to itself (e.g., the
current page appears in the navigation), the prefetch hint for that
URL is skipped. The current page's URL is available from `PageMeta`
as `current_url` (e.g., `/about.html`). When fragments are enabled,
the self-reference check must compare against the corresponding
fragment path (`/_fragments/about.html`). The comparison derives the
fragment path from `current_url` using the same logic as
`compute_fragment_path` -- prepend `/{fragment_dir}/` to the path
with the leading `/` stripped. This logic is simple enough to inline
(no need to import the private function from `template::functions`).

Index pages require special handling: both `/index.html` and `/`
should be recognized as the same page. The self-reference check
normalizes paths by treating a trailing `/` as `/index.html`.

### Interaction with Fragments Disabled

When `build.fragments = false`, the `link_to()` function emits plain
`href` attributes without `hx-get`. In this mode, navigation prefetch
still works but targets full page URLs extracted from `<a href>`
attributes instead. The prefetch is `<link rel="prefetch" as="document">`
since the browser will load a full HTML page on click.

When scanning `<a href>` attributes (fragments disabled mode), we only
include links with local paths (starting with `/`). Links with
`target="_blank"` are excluded since they open in a new tab and the
user experience benefit of prefetching is lower.

### Pipeline Position

The current build pipeline (after this feature) becomes:

```
render template
  -> strip fragment markers
  -> localize assets
  -> optimize images (img -> picture)
  -> rewrite CSS background images
  -> plugin post_render_html
  -> critical CSS inlining
  -> preload/prefetch hints  <-- NEW
  -> minify HTML
  -> write to disk
```

**Rationale for this position:**

- Must run after image optimization so that `<img>` tags have been
  rewritten to `<picture>` elements with `<source srcset>`. The
  auto-detection approach reads variant information directly from
  these `<source>` elements rather than re-scanning the filesystem.
- Must run after critical CSS inlining because critical CSS may inject
  `<link rel="preload" as="style">` for deferred stylesheets; we want
  to avoid duplicate or conflicting preload hints for stylesheets.
- Must run after plugin hooks because plugins may inject additional
  images or links.
- Must run before minification so the injected `<link>` tags benefit
  from `minify-html`'s optimization.

**Fragments do NOT get preload/prefetch hints.** Fragments are partial
HTML snippets loaded via HTMX. They do not have a `<head>` element,
so there is no place to inject `<link>` tags. Resource hints only
apply to full page renders.

### Dev Server

Preload/prefetch hints are **disabled during dev server** builds.
The dev server's render functions (`render_static_page_dev`,
`render_dynamic_page_dev` in `src/dev/rebuild.rs`) have their own
pipeline that skips image optimization, critical CSS, and
minification. Resource hints are similarly not called from these
functions. No code changes are needed in `src/dev/rebuild.rs` --
the hint injection only happens in `render_static_page` and
`render_dynamic_page` in `src/build/render.rs`.

### Configuration

Add to `src/config/mod.rs`:

```rust
/// Configuration for preload and prefetch resource hints.
///
/// Located under `[build.hints]` in site.toml.
#[derive(Debug, Clone, Deserialize)]
pub struct HintsConfig {
    /// Master switch for resource hints. Default: true.
    /// When false, no preload or prefetch hints are generated.
    #[serde(default = "default_true")]
    pub enabled: bool,

    /// Whether to auto-detect the hero image from rendered HTML when
    /// no `hero_image` is set in frontmatter. Default: true.
    ///
    /// When enabled, the first `<img>` tag without `loading="lazy"`
    /// is treated as the hero image and gets a preload hint.
    #[serde(default = "default_true")]
    pub auto_detect_hero: bool,

    /// Whether to generate prefetch hints for navigation links.
    /// Default: true.
    #[serde(default = "default_true")]
    pub prefetch_links: bool,

    /// Maximum number of `<link rel="prefetch">` hints per page.
    /// Prevents excessive prefetching on pages with many links.
    /// Default: 5.
    #[serde(default = "default_max_prefetch")]
    pub max_prefetch: usize,

    /// Fallback value for the `imagesizes` attribute on hero image
    /// preload hints, used when the hero image has responsive variants
    /// but no `sizes` attribute can be read from the `<img>` tag.
    /// Default: "100vw".
    ///
    /// When auto-detecting from a `<picture>` element, the `sizes`
    /// attribute on the inner `<img>` tag is preferred over this
    /// config value. This config is the fallback for frontmatter
    /// `hero_image` paths that have no corresponding `<img>` in HTML
    /// (e.g., CSS background images with filesystem variant lookup).
    #[serde(default = "default_image_sizes")]
    pub hero_image_sizes: String,

    /// Glob patterns for link hrefs to exclude from prefetching.
    /// Useful for excluding large pages or specific sections.
    /// Matched against the hx-get or href value using
    /// `glob::Pattern` (same as image exclude patterns).
    #[serde(default)]
    pub exclude_prefetch: Vec<String>,
}

fn default_max_prefetch() -> usize {
    5
}

fn default_image_sizes() -> String {
    "100vw".to_string()
}

impl Default for HintsConfig {
    fn default() -> Self {
        Self {
            enabled: true,
            auto_detect_hero: true,
            prefetch_links: true,
            max_prefetch: default_max_prefetch(),
            hero_image_sizes: default_image_sizes(),
            exclude_prefetch: Vec::new(),
        }
    }
}
```

Add to `BuildConfig`:

```rust
pub struct BuildConfig {

    //     oob_blocks, minify, critical_css) ...

    /// Preload/prefetch resource hints configuration.
    #[serde(default)]

}
```

Update the `Default` impl for `BuildConfig`:

```rust
impl Default for BuildConfig {
    fn default() -> Self {
        Self {
            // ... existing fields ...
            critical_css: CriticalCssConfig::default(),
            hints: HintsConfig::default(),
        }
    }
}
```

**Default enabled rationale:** Unlike critical CSS which is opt-in
(because it changes how stylesheets load), resource hints are purely
additive -- they add `<link>` tags but do not modify or remove any
existing content. They cannot break anything. Defaulting to enabled
means users get the performance benefit without needing to know the
feature exists.

### Internal Types

These types live in `src/build/hints/` and are not public API:

```rust
/// A resource hint to be injected into the page's `<head>`.
enum ResourceHint {
    /// `<link rel="preload">` for the hero image.
    HeroPreload {
        /// The image URL path (e.g. "/assets/hero.jpg").
        href: String,
        /// MIME type of the image (e.g. "image/jpeg").
        mime_type: String,
        /// If the image has responsive variants, the srcset string
        /// for the `imagesrcset` attribute.
        imagesrcset: Option<String>,
        /// The `imagesizes` value (e.g. "100vw").
        imagesizes: Option<String>,
    },
    /// `<link rel="prefetch">` for a likely next navigation.
    NavigationPrefetch {
        /// The fragment or page URL to prefetch.
        href: String,
        /// Whether this targets a full document (vs a fragment).
        /// When true, adds `as="document"` to the `<link>` tag.
        is_document: bool,
    },
}
```

The `ResourceHint` enum is used internally to collect hints before
serializing them into HTML. It is not exposed outside the module.

### Hero Image Resolution

When the hero image has been processed by image optimization, the
`<img>` tag will have been rewritten to a `<picture>` element with
`<source>` elements. The preload hint needs to match this:

```html
<!-- For a hero image with responsive variants: -->
<link rel="preload" as="image" href="/assets/hero.jpg"
      imagesrcset="/assets/hero-480w-a1b2c3d4.avif 480w, /assets/hero-768w-a1b2c3d4.avif 768w, /assets/hero-1200w-a1b2c3d4.avif 1200w"
      imagesizes="100vw"
      type="image/avif">

<!-- For a hero image without responsive variants: -->
<link rel="preload" as="image" href="/assets/hero.jpg"
      type="image/jpeg">
```

The resolution process:

1. Start with the `hero_image` path from frontmatter (or auto-detected
   `<img>` src).
2. Search the rendered HTML for a `<picture>` element containing an
   `<img>` with a matching `src` attribute.
3. If a matching `<picture>` is found, extract the `srcset` and `type`
   from the first `<source>` child (which is the best format -- AVIF
   if available, per the image optimization module's format ordering
   in `determine_format_order`). If the `<img>` has a `sizes`
   attribute, use it for `imagesizes`; otherwise fall back to the
   `hero_image_sizes` config value.
4. If no matching `<picture>` is found, scan dist_dir for variant
   files matching `{stem}-*w-*.{format}` in the same directory as
   the original image. This handles the case where `hero_image` is
   a CSS background image with no corresponding `<img>` tag. If
   variants are found, build `imagesrcset` from the best format
   available (prefer AVIF, then WebP, then original).
5. If no variants exist at all, generate a simple preload with just
   `href` and `type` (inferred from file extension).

**Why prefer AVIF in preload but not WebP?** The preload hint can only
specify one format. The `<picture>` element handles format negotiation
via multiple `<source>` tags, but `<link rel="preload">` fetches exactly
one resource. We pick the format that offers the best compression (AVIF)
and set `type="image/avif"` so browsers that don't support AVIF will
ignore the hint and discover the image through normal `<picture>`
parsing. This is the behavior recommended by web.dev.

For browsers that ignore the preload (because they don't support the
hinted format), there is no penalty -- they simply discover the image
later via normal parsing, which is the baseline behavior without this
feature.

## API Surface

### Public Function

The module `build::hints` exposes a single public function:

```rust
/// Inject preload and prefetch resource hints into HTML.
///
/// This is the main entry point called from the build pipeline.
/// It is infallible by design: any internal error causes a fallback
/// to returning the original HTML unchanged, with a warning logged.
///
/// Parameters:
/// - `html`: The rendered HTML string (after image optimization and
///   critical CSS inlining).
/// - `config`: The hints configuration from site.toml.
/// - `dist_dir`: Path to the dist/ directory for checking image variants.
/// - `hero_image`: Optional hero image path from frontmatter.
/// - `current_url`: The URL path of the current page (e.g. "/about.html"),
///   used for self-link exclusion in prefetch.
/// - `fragment_dir`: The fragment directory name (e.g. "_fragments"),
///   used for self-link detection.
/// - `fragments_enabled`: Whether the HTMX fragment system is active.
///
/// Returns the (possibly rewritten) HTML string.
pub fn inject_resource_hints(
    html: &str,
    config: &HintsConfig,
    dist_dir: &Path,
    hero_image: Option<&str>,
    current_url: &str,
    fragment_dir: &str,
    fragments_enabled: bool,
) -> String
```

**Return type rationale:** Same as `inline_critical_css` -- resource
hints are a pure optimization. If anything goes wrong, the correct
behavior is to return the original HTML unchanged. Making the return
type infallible ensures callers cannot accidentally propagate an error
that breaks the build.

**Why `fragment_dir` as a parameter:** The self-referencing link check
needs to know how to construct the current page's fragment URL from
`current_url`. Rather than passing the entire `BuildConfig`, we pass
just the `fragment_dir` string. This keeps the API minimal and avoids
coupling the hints module to the full config structure.

### Internal Functions

```rust
/// Determine the hero image for a page.
///
/// Priority:
/// 1. Explicit `hero_image` from frontmatter.
/// 2. Auto-detected first `<img>` without `loading="lazy"` (if
///    `auto_detect_hero` is enabled).
/// 3. None (no preload hint generated).
fn resolve_hero_image(
    html: &str,
    frontmatter_hero: Option<&str>,
    auto_detect: bool,
) -> Option<String>

/// Build the hero preload hint by extracting responsive variant
/// information from the rendered HTML's `<picture>` elements.
///
/// First checks for a `<picture>` element with a matching `<img src>`.
/// If found, extracts `srcset` and `type` from the first `<source>`.
/// If not found, falls back to scanning dist_dir for variant files.
fn build_hero_preload(
    hero_src: &str,
    html: &str,
    dist_dir: &Path,
    image_sizes: &str,
) -> ResourceHint

/// Scan dist_dir for responsive variants of an image.
///
/// Looks for files matching `{stem}-*w-*.{format}` in the same
/// directory as the original. Groups by format, prefers AVIF,
/// then WebP, then original.
///
/// This is the fallback path when no `<picture>` element is found
/// in the HTML (e.g., hero image is a CSS background-image).
fn scan_dist_for_variants(
    hero_src: &str,
    dist_dir: &Path,
) -> Option<(String, Vec<(String, u32)>)>  // (mime_type, [(url, width)])

/// Scan HTML for `hx-get` (or `href` if fragments disabled) attributes
/// and collect unique local URLs for prefetching.
///
/// Excludes:
/// - External URLs (starting with http:// or https://)
/// - The current page's own fragment URL (or full URL if fragments
///   disabled)
/// - URLs matching exclude patterns (via `glob::Pattern`)
/// - Anchor-only links (#...)
/// - Links with `target="_blank"` (only checked in href-scanning mode)
///
/// Returns at most `max_prefetch` URLs in document order.
fn collect_prefetch_urls(
    html: &str,
    current_url: &str,
    fragment_dir: &str,
    fragments_enabled: bool,
    max_prefetch: usize,
    exclude_patterns: &[String],
) -> Vec<String>

/// Serialize a list of ResourceHint values into HTML `<link>` tag
/// strings.
fn hints_to_html(hints: &[ResourceHint]) -> String

/// Inject the hint `<link>` tags into the HTML `<head>`.
///
/// Uses lol_html to find the opening `<head>` tag and prepend the
/// hint tags as the first children. This is simpler and more
/// reliable than tracking the "last <meta> tag" position in a
/// streaming rewriter.
///
/// Placement as early `<head>` children is intentional: preload
/// hints should appear as early as possible so the browser's
/// preload scanner discovers them immediately.
fn inject_into_head(
    html: &str,
    hint_html: &str,
) -> Result<String, String>
```

### Frontmatter Threading

The `hero_image` field from frontmatter must be threaded through the
build pipeline to reach `inject_resource_hints`. The existing pipeline
already passes config and dist_dir; we add `hero_image` as a parameter.

In `render_static_page` and `render_dynamic_page`:

```rust
// Preload/prefetch hints (after critical CSS, before minify).
let full_html = if config.build.hints.enabled {
    hints::inject_resource_hints(
        &full_html,
        &config.build.hints,
        dist_dir,
        page.frontmatter.hero_image.as_deref(),
        &url_path,
        &config.build.fragment_dir,
        config.build.fragments,
    )
} else {
    full_html
};
```

For dynamic pages, the `hero_image` may reference template variables
like `{{ post.cover_image }}`. However, since frontmatter is parsed
before rendering and `hero_image` is a static string, template
interpolation is not available in frontmatter fields. This is
acceptable because:

1. Dynamic pages with varying hero images can rely on auto-detection,
   which scans the rendered HTML.
2. If a dynamic template has a consistent hero image (e.g., a category
   header), the frontmatter `hero_image` field works.

This is a deliberate simplification. Supporting template interpolation
in frontmatter would require rendering the frontmatter field before
rendering the template, adding complexity for a marginal benefit.

### Auto-Detection Details

When `auto_detect_hero` is enabled and no frontmatter `hero_image` is
set, the function scans rendered HTML using `lol_html`:

1. Find all `<img>` tags.
2. Skip images with `loading="lazy"` (these are below-the-fold by
   definition -- note that image optimization does not automatically
   set `loading="lazy"`, so only images explicitly marked lazy in
   templates are skipped).
3. Skip images with `width` or `height` attributes where both are
   less than 100px (icons, spacers).
4. Skip images with `role="presentation"` or empty `alt=""` (decorative
   images unlikely to be the hero).
5. Return the `src` of the first qualifying `<img>`.

If the image is inside a `<picture>` element (because image optimization
rewrote it), we still use the `<img>` `src` as the base URL for
variant detection. The `<source>` elements provide the information
needed to build `imagesrcset`.

**`<picture>` detection nuance with `lol_html`:** Since `lol_html` is a
streaming rewriter (not a tree parser), we cannot directly check "is
this `<img>` inside a `<picture>`?". Instead, `build_hero_preload`
handles this by searching the full HTML for a `<picture>` element
containing the hero image's `src`. This second scan is acceptable
because it only runs once per page for the single hero image.

When auto-detecting from a `<picture>` element, we extract the
`imagesrcset` and `type` from the first `<source>` child, which is
the best format (AVIF if available, per the image optimization
module's ordering in `determine_format_order` in
`src/assets/html_rewrite.rs`). This avoids re-scanning dist_dir for
variants that are already encoded in the HTML.

## Error Handling

| Scenario | Behavior |
|---|---|
| Hero image path not found in dist_dir | Log debug (may be external URL or missing file), generate simple preload without `imagesrcset` |
| No `<head>` element in HTML | Log warning, return original HTML unchanged |
| No qualifying `<img>` found (auto-detect) | No hero preload hint generated; prefetch hints still generated |
| No `hx-get` or `href` links found | No prefetch hints generated; hero preload still generated |
| lol_html rewriting error | Log warning, return original HTML unchanged |
| Invalid hero_image path in frontmatter | Log warning with template name, skip hero preload |
| Image variant scan fails (IO error) | Log debug, fall back to simple preload without `imagesrcset` |
| All prefetch candidates excluded | No prefetch hints generated (no-op) |
| `max_prefetch = 0` | No prefetch hints generated; hero preload still works |
| No `<picture>` found for frontmatter hero_image | Fall back to dist_dir variant scan; if that also fails, generate simple preload |
| `<source>` has no `srcset` attribute | Skip responsive preload, generate simple preload |
| `glob::Pattern::new` fails for exclude pattern | Log warning, skip that pattern, continue with remaining |

**Principle:** Same as critical CSS -- resource hints are a pure
optimization. Failures must never break the build or degrade the
output. Every error path falls back to returning the original HTML
unchanged.

## Edge Cases and Failure Modes

### Responsive hero image preload

When image optimization generates responsive variants, the preload
hint must use `imagesrcset` rather than a bare `href`. A bare `href`
preload for a responsive image wastes bandwidth because the browser
may preload a size it won't use.

If the hero image has variants but we cannot determine the correct
format or sizes, we fall back to a simple preload with just the
original image `href`. This is strictly better than no preload at
all.

### Hero image set via CSS background-image

If the hero image is a CSS `background-image` rather than an `<img>`
tag, auto-detection will not find it. The user must use the frontmatter
`hero_image` field. In this case, there is no `<picture>` element in
the HTML, so `build_hero_preload` falls back to scanning dist_dir for
variant files matching `{stem}-*w-*.{format}`. If the CSS background
image was also processed by `rewrite_css_background_images`, the
variants will exist on disk. This is documented and expected.

### Duplicate preload hints

If the critical CSS module already generated a
`<link rel="preload" as="style">` for a stylesheet, our module may
also be scanning `<link>` tags. We only generate preload hints for
images (`as="image"`), never for stylesheets, so there is no
conflict.

If the user manually adds a `<link rel="preload">` for the hero
image in their template, we may generate a duplicate. To prevent
this, the collection pass scans existing `<link rel="preload">`
tags in `<head>` and skips generating a hint if one already exists
for the same `href`.

### Pages without `<head>`

Fragment files and malformed HTML may lack a `<head>` element. The
feature is designed to only operate on full page renders (not
fragments), but as a safety measure, `inject_into_head` gracefully
returns the original HTML if no `<head>` is found.

### Self-referencing navigation links

A page's navigation often includes a link to the current page itself
(e.g., highlighted "active" nav item). Prefetching the current page's
own fragment is wasteful. We derive the current page's fragment URL
from `current_url` and `fragment_dir`:

```rust
// For current_url = "/about.html" and fragment_dir = "_fragments":
// -> self_fragment_url = "/_fragments/about.html"
let clean = current_url.trim_start_matches('/');
let self_fragment_url = format!("/{}/{}", fragment_dir, clean);
```

Each candidate `hx-get` URL is compared against `self_fragment_url`.
When fragments are disabled, we compare against `current_url` directly.

**Index page normalization:** For index pages, `current_url` is
`/index.html`, but navigation links may use `/`. The self-reference
check normalizes by treating a URL of `/` as `/index.html` before
comparison. Similarly, paths like `/posts/` are treated as
`/posts/index.html`.

### External links in navigation

Links with `hx-get` pointing to external URLs (http/https) are
skipped. The `hx-get` attribute can technically point anywhere,
but in eigen sites, it always points to local fragment files. If
a user somehow generates an external `hx-get`, skipping it is the
correct behavior since cross-origin prefetch has different
semantics and security implications.

### Anchor-only links

Links with `href="#section"` or `hx-get="#section"` (anchor-only,
no path) are skipped. These are in-page navigation, not separate
resources.

### Interaction with content hashing (future)

The proposals doc mentions content hashing. When implemented, hero
image paths and fragment paths will include content hashes. Since
the hints module reads variant URLs directly from the rendered HTML
(which will contain hashed paths), no special handling is needed.

### Very large navigation pages

A page with hundreds of links (e.g., a sitemap page, archive index)
should not generate hundreds of prefetch hints. The `max_prefetch`
config (default 5) handles this. The limit applies after deduplication
and exclusion filtering.

### Interaction with `hx-boost`

If a user uses `hx-boost` instead of `link_to()`, the `<a>` tags
will not have `hx-get` attributes. In this case, the prefetch system
will not find any `hx-get` URLs.

**Decision:** We do NOT attempt to detect `hx-boost` and reconstruct
fragment URLs from `<a href>`. This was considered but rejected
because:

1. `hx-boost` is a client-side HTMX behavior that can be configured
   in many ways (body-level, container-level, per-link). Detecting
   all configurations reliably from static HTML is fragile.
2. Reconstructing fragment URLs requires duplicating the
   `compute_fragment_path` logic from `template::functions`, which
   is currently private and handles block names.
3. Users who want prefetch hints with `hx-boost` can switch to
   `link_to()`, which is the recommended approach for eigen sites
   and provides explicit `hx-get` attributes.

If `hx-boost` support becomes needed, it would be better addressed
by adding an HTMX extension that does client-side prefetching based
on viewport intersection, which is out of scope for this build-time
feature.

### Preload hint for `<picture>` with multiple formats

When the hero is a `<picture>` element, we generate the preload for
the best format only (typically AVIF). We do NOT generate multiple
preload hints for each format because:

1. The browser would fetch all of them, wasting bandwidth.
2. The `type` attribute on `<link rel="preload">` tells the browser
   to skip unsupported formats, so a single AVIF preload is safe
   even for browsers that don't support AVIF.

### Interaction with `fetchpriority`

The `fetchpriority="high"` attribute on `<img>` tags and
`<link rel="preload">` is a newer API that tells the browser to
prioritize fetching a resource. This spec does not add
`fetchpriority` to the preload hint or the hero `<img>` tag.
However, if `fetchpriority` support is added in the future, the
preload hint should include `fetchpriority="high"` and the hero
`<img>` tag should also get `fetchpriority="high"` (which would
require an additional `lol_html` rewrite step). This is noted here
for future reference.

## What Is NOT In Scope

1. **Intersection Observer-based prefetching.** Client-side JS that
   prefetches links as they scroll into view. This is a different
   approach (client-side vs build-time) and belongs in a separate
   feature or HTMX extension. We generate static `<link>` tags in
   the HTML.

2. **Preloading fonts.** Font preloading is best done manually in
   templates because font selection is a design decision. Auto-detecting
   which fonts to preload would require parsing CSS, which critical CSS
   already does. If font preloading is desired, it should be a separate
   feature that integrates with critical CSS.

3. **Preloading JavaScript.** Eigen sites use HTMX (loaded from CDN)
   and no other JS. Preloading the HTMX script could be done manually
   with a `<link rel="preload">` in the base template.

4. **Predictive prefetching.** Analyzing user behavior to predict likely
   next navigations. This requires analytics data that is not available
   at build time.

5. **Per-page prefetch overrides.** A frontmatter field to customize
   which links to prefetch on a specific page. The site-level config is
   sufficient. Per-page `hero_image` is supported because it is tied
   to content; per-page prefetch control would add complexity for
   minimal benefit.

6. **Priority hints (`fetchpriority`).** The `fetchpriority` attribute
   on `<link>` and `<img>` is a newer API. It could be added later as
   an enhancement but is not part of this initial implementation (see
   "Interaction with `fetchpriority`" in Edge Cases for notes).

7. **Preloading fragment files for the current page.** The current
   page's fragments are only fetched if the user navigates back to it,
   which is handled by browser cache or HTMX cache.

8. **DNS prefetch / preconnect for external resources.** These are
   different hint types that could be a separate feature.

9. **`hx-boost` detection and fragment URL reconstruction.** Too fragile
   for build-time static analysis. Users should use `link_to()` for
   reliable prefetch hint generation (see "Interaction with `hx-boost`"
   in Edge Cases for rationale).

## Module Structure

```
src/build/
  hints/
    mod.rs          -- public API (inject_resource_hints),
                       orchestration, hero image resolution,
                       shared types (ResourceHint), inject_into_head
    preload.rs      -- hero image preload: <picture>/<source> extraction,
                       dist_dir variant scanning, imagesrcset building,
                       preload tag generation
    prefetch.rs     -- navigation prefetch: hx-get/href scanning,
                       URL deduplication and filtering, self-reference
                       detection, prefetch tag generation
```

Three files because the concerns are distinct:
- `mod.rs`: orchestration, shared types, `inject_into_head` (~100 lines)
- `preload.rs`: hero image detection and variant resolution (~150 lines)
- `prefetch.rs`: link scanning and filtering (~120 lines)

This follows the pattern established by `src/build/critical_css/`
which has `mod.rs`, `extract.rs`, `rewrite.rs`.

## Integration Points

### build/render.rs

In `render_static_page` and `render_dynamic_page`, add the hints step
after critical CSS inlining and before minification. Both functions
already have access to `page.frontmatter`, `config`, `dist_dir`, and
`url_path`:

```rust
// Preload/prefetch hints (after critical CSS, before minify).
let full_html = if config.build.hints.enabled {
    hints::inject_resource_hints(
        &full_html,
        &config.build.hints,
        dist_dir,
        page.frontmatter.hero_image.as_deref(),
        &url_path,
        &config.build.fragment_dir,
        config.build.fragments,
    )
} else {
    full_html
};
```

Note: No changes needed to `render_static_page` or `render_dynamic_page`
function signatures. The `page.frontmatter.hero_image` field is already
available via `page`, and the hints config is available via `config`.
This is a key advantage over critical CSS, which required threading a
`StylesheetCache` through the function signatures.

### build/mod.rs

Add `pub mod hints;` to the module declarations:

```rust
pub mod context;
pub mod critical_css;
pub mod fragments;
pub mod hints;         // <-- NEW
pub mod minify;
pub mod output;
pub mod render;
pub mod sitemap;
```

### frontmatter/mod.rs

Add `hero_image` field to `Frontmatter` and `RawFrontmatter`:

```rust
pub struct Frontmatter {
    // ... existing fields (collection, slug_field, item_as,
    //     data, fragment_blocks) ...
    pub hero_image: Option<String>,
}

#[derive(Debug, Deserialize)]
struct RawFrontmatter {
    // ... existing fields ...
    hero_image: Option<String>,
}
```

Update `parse_frontmatter` to map the field:

```rust
Ok(Frontmatter {
    // ... existing fields ...
    hero_image: raw.hero_image,
})
```

Update `Frontmatter::default()`:

```rust
impl Default for Frontmatter {
    fn default() -> Self {
        Self {
            // ... existing fields ...
            hero_image: None,
        }
    }
}
```

### config/mod.rs

Add `HintsConfig` struct and `hints` field to `BuildConfig`. Reuse
the existing `default_true` function. Update the `Default` impl for
`BuildConfig`.

### No new Cargo.toml dependencies

All required crates are already dependencies:
- `lol_html` for HTML rewriting (streaming scan + inject)
- `glob` for exclude pattern matching (same as image exclude)
- `regex` for URL/srcset extraction if needed

The `image` crate is NOT used by this module. MIME types are inferred
from file extensions using a simple match (consistent with
`guess_format_from_ext` in `assets::images`).
