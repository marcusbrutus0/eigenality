# Plan: Extract Shared Post-Render Pipeline (eigen-855)

## Problem

`render_static_page` and `render_dynamic_page` (in `src/build/render.rs`) share an identical
~140-line post-render pipeline. The same duplication exists between `render_static_page_dev`
and `render_dynamic_page_dev` (in `src/dev/rebuild.rs`). Any bug fix or new step must be
applied in four places.

## Goal

Extract two helper functions:

- `finalize_page_html` â€” prod pipeline, lives in `src/build/render.rs`
- `finalize_page_html_dev` â€” dev pipeline, lives in `src/dev/rebuild.rs`

Each function also absorbs the fragment pipeline, which is also duplicated across the same
four call sites.

## Key observations from reading the code

### `inject_status_banner` is internally guarded

`src/dev/inject.rs:51-53`: The function returns early if `label.is_empty()`. Dynamic dev pages
can pass `""` and the call is a safe no-op. No caller-side guard needed.

### Dev `localize_fragments_dev` is private and called from exactly two places

Both call sites are in `rebuild.rs`. It can be inlined into `finalize_page_html_dev` and
the standalone function deleted.

### Fragment filter differs between static and dynamic (dev only)

- Static dev: filters by `page.frontmatter.fragment_blocks` (Some â†’ filter, None â†’ keep all)
- Dynamic dev: same filter
- Both use `fragments::extract_fragments(&rendered)` (not `extract_page_fragments`)

### Prod static uses `extract_page_fragments`; prod dynamic also uses it

`extract_page_fragments` takes `page: &PageDef` and `content_block`. Both prod call sites
pass the same args. This can be called inside `finalize_page_html`.

### Fragment block filter lives outside the finalize call in dev

In dev, the fragment filter (`fragment_blocks`) is applied after `extract_fragments`. The
finalize function needs access to `page.frontmatter.fragment_blocks` to replicate this.

### Error label context

Prod: error context strings embed `tmpl_name` for static, `tmpl_name + slug` for dynamic.
The finalize function needs a `label: &str` parameter pre-formatted by the caller (e.g.,
`"template 'foo.html'"` or `"template 'foo.html' slug 'bar'"`).

### Source asset URLs

The prod pipeline needs `source_asset_urls: &HashSet<String>` for both `localize_assets` and
`localize_fragments`. Dynamic pages compute this per-item. Static pages compute it once after
drain. Both are already available at the call site.

Dev pipelines do not use source assets â€” they always pass an empty `HashSet`.

## Proposed signatures

### Prod (in `src/build/render.rs`)

```rust
struct FinalizeInput<'a> {
    rendered: &'a str,                          // raw rendered output (pre-strip)
    output_path: &'a Path,
    url_path: &'a str,
    label: &'a str,                             // "template 'foo'" or "template 'foo' slug 'bar'"
    page: &'a PageDef,
    resolved_seo: ResolvedSeo,
    resolved_schema: ResolvedSchema,
    source_requests: Vec<SourceAssetRequest>,   // drained from ctx.source_asset_collector before call
    source_asset_urls: HashSet<String>,         // URL set derived from source_requests for localize_assets / localize_fragments
    block_names: Vec<String>,                   // pre-computed before strip if view_transitions enabled
}

async fn finalize_page_html(
    input: FinalizeInput<'_>,
    ctx: &BuildContext,
) -> Result<()>
```

Returns `()` because writing to disk is the final side effect. The caller only needs
`RenderedPage`, which it constructs from `url_path` and `output_path` it already has.

### Dev (in `src/dev/rebuild.rs`)

```rust
struct DevFinalizeInput<'a> {
    rendered: &'a str,           // raw rendered output (pre-strip)
    output_path: &'a Path,
    url_path: &'a str,
    label: &'a str,              // "template 'foo'" or "template 'foo' slug 'bar'"
    page: &'a PageDef,
    draft_label: &'a str,        // status banner label (empty string for dynamic)
}

async fn finalize_page_html_dev(
    input: DevFinalizeInput<'_>,
    ctx: &mut DevRenderContext<'_>,
) -> Result<()>
```

## Pipeline steps mapped to each function

### `finalize_page_html` (prod)

1. `block_names` â€” caller pre-computes before strip if `view_transitions.enabled`
2. `strip_fragment_markers(rendered)` â†’ `full_html`
3. Lock `asset_cache` â†’ `localize_assets(&full_html, ..., &input.source_asset_urls)` â†’ `resolve_source_assets(&html, &input.source_requests, ...)` â†’ unlock
4. `optimize_and_rewrite_images`
5. `rewrite_css_background_images`
6. `plugin_registry.post_render_html`
7. If `critical_css.enabled`: lock `css_cache` â†’ `inline_critical_css` â†’ drop lock
8. If `hints.enabled`: `inject_resource_hints`
9. `seo::inject_seo_tags`
10. `json_ld::inject_json_ld`
11. If `view_transitions.enabled`: `inject_view_transitions(&full_html, &block_names)`
12. If `minify`: `minify_html`
13. If `analytics` configured: `inject_analytics`
14. `create_dir_all` + `fs::write` to `dist_dir/output_path`
15. If `fragments` enabled:
    a. `extract_page_fragments(rendered, page, &config.build.content_block)`
    b. If non-empty: `localize_fragments` â†’ `optimize_fragment_images` â†’ optionally `minify_fragments` â†’ `write_fragments`

Note: `block_names` is computed by the caller before calling `finalize_page_html` so the
`rendered` string (with markers) is available. Pass it in via `FinalizeInput`.

### `finalize_page_html_dev` (dev)

1. `strip_fragment_markers(rendered)` â†’ `full_html`
2. `localize_assets(&full_html, ..., &HashSet::new())`
3. `plugin_registry.post_render_html`
4. `inject_reload_script`
5. `inject_status_banner(&full_html, draft_label)` â€” no-op if `draft_label` is `""`
6. `create_dir_all` + `fs::write` to `dist_dir/output_path`
7. If `fragments` enabled:
    a. `extract_fragments(rendered)` â†’ filter by `page.frontmatter.fragment_blocks`
    b. If non-empty: inline `localize_fragments_dev` logic â†’ `write_fragments`

## Implementation steps

### Step 1: Prod pipeline extraction (`src/build/render.rs`)

1. Define `FinalizeInput<'a>` struct before `finalize_page_html`.
2. Write `async fn finalize_page_html(input: FinalizeInput<'_>, ctx: &BuildContext) -> Result<()>`.
   - Copy pipeline from `render_static_page` lines 467-646 verbatim, substituting struct fields
     for the local variables that the struct now carries.
   - For error context strings: use `input.label` (e.g., `wrap_err_with(|| format!("... for '{}'", input.label))`).
   - For `resolved_seo` and `resolved_schema`: these are consumed (not borrowed), so move them
     into the struct.
3. In `render_static_page`:
   - Pre-compute `block_names` (lines 468-472) as before â€” keep outside finalize because it
     needs `rendered` before strip.
   - Drain source asset collector (lines 476-483) â€” keep outside finalize because it uses `ctx.source_asset_collector`.
     Produce both `page_source_requests: Vec<SourceAssetRequest>` and `source_asset_urls: HashSet<String>` as today.
   - Build `FinalizeInput` (pass `source_requests: page_source_requests` and `source_asset_urls`) and call `finalize_page_html(input, ctx).await?`.
   - Return `Ok(RenderedPage { ... })` using already-local `url_path`, `is_index`.
4. In `render_dynamic_page` (inside item loop):
   - Same pattern: pre-compute `block_names`, drain collector (both `source_requests` and `source_asset_urls`), build `FinalizeInput`, call finalize.
   - The label is `format!("template '{}' slug '{}'", tmpl_name, slug)`.
   - Return `RenderedPage` from already-local `url_path`.

### Step 2: Dev pipeline extraction (`src/dev/rebuild.rs`)

1. Define `DevFinalizeInput<'a>` struct.
2. Write `async fn finalize_page_html_dev(input: DevFinalizeInput<'_>, ctx: &mut DevRenderContext<'_>) -> Result<()>`.
   - Inline the `localize_fragments_dev` logic directly inside the fragment block (no longer a
     separate function).
   - `inject_status_banner` is always called; passes `input.draft_label` (empty string for
     dynamic pages is a safe no-op).
3. In `render_static_page_dev`:
   - Compute `draft_label` as before.
   - Build `DevFinalizeInput` (with `draft_label: &draft_label`) and call `finalize_page_html_dev`.
4. In `render_dynamic_page_dev` (inside item loop):
   - Pass `draft_label: ""` in `DevFinalizeInput`.
5. Delete `localize_fragments_dev` â€” now inlined inside `finalize_page_html_dev`.

## Risks / watch-outs

- **`source_requests` and `source_asset_urls` ownership**: The caller drains
  `ctx.source_asset_collector` into `page_source_requests: Vec<SourceAssetRequest>` and
  derives `source_asset_urls: HashSet<String>` from it (as today). Both are moved into
  `FinalizeInput`. Inside `finalize_page_html`, `source_requests` is passed by reference to
  `resolve_source_assets` and `source_asset_urls` is borrowed by both `localize_assets` and
  `localize_fragments`.
- **`rendered` lifetime in finalize**: `FinalizeInput.rendered: &'a str` borrows from the
  local `rendered: String` in the caller. The struct lifetime `'a` is tied to the call frame â€”
  this works because `finalize_page_html` is called and awaited before `rendered` is dropped.
- **`block_names` pre-computation**: must happen before `finalize_page_html` because it needs
  the raw `rendered` string with markers. The caller keeps this responsibility.
- **`resolved_seo` / `resolved_schema` are consumed**: move them into `FinalizeInput` rather
  than borrowing, since they are not used after the call.
- **Error messages**: the prod pipeline uses per-field `wrap_err_with` with `tmpl_name` and
  optionally `slug`. Using a pre-formatted `label: &str` from the caller keeps the messages
  identical to before.
- **`extract_page_fragments` vs `extract_fragments`**: prod uses `extract_page_fragments`
  (filters by `content_block`); dev uses `extract_fragments`. The finalize functions use the
  correct one for their context.
- **Fragment filter in dev**: after `extract_fragments`, dev filters by
  `page.frontmatter.fragment_blocks`. Pass `page: &'a PageDef` into `DevFinalizeInput` so
  `finalize_page_html_dev` can apply this filter internally.
